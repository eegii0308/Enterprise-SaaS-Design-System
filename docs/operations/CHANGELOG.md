## 2026-07-11

### Added

- Implemented Phase 9B password reset, reusing the invitation infrastructure from Phase 9A rather than duplicating it: `lib/security/tokens.ts`'s `generateInvitationToken`/`hashInvitationToken` were renamed to `generateSecureToken`/`hashSecureToken` and are now shared, unchanged, by both `lib/invitations/management.ts` and the new `lib/auth/password-reset.ts`. A new `escapeHtml` helper (`lib/email/html.ts`) was extracted out of `lib/email/templates/invitation.ts` (previously private to that file) and is now shared with the new `lib/email/templates/password-reset.ts`.
- Added `PasswordResetToken` (`prisma/migrations/20260711010000_phase_9b_password_reset`): `userId`, unique `tokenHash`, `expiresAt`, nullable `usedAt`. Deliberately simpler than `Invitation` (no status enum, no admin-facing list/cancel/resend UI) since password reset has no lifecycle beyond issue-and-consume.
- `requestPasswordReset` (`lib/auth/password-reset.ts`) looks up the user by email, and if found and `ACTIVE`, invalidates any earlier outstanding reset tokens for that user (`usedAt` set), creates a new one with a 1-hour expiry (vs. the 7-day invitation TTL -- this token grants control over an existing account, so its exposure window is deliberately short), and writes a `PASSWORD_RESET_REQUESTED` audit log. Returns `{ sent: false }` for an unknown or non-`ACTIVE` email without throwing or otherwise varying its result shape in a way a caller could use to distinguish the two cases from timing/branching alone.
- `forgotPasswordAction` (`lib/auth/actions.ts`) now actually calls `requestPasswordReset` and sends the email (previously a stub that validated the email and returned success without doing anything). The entire body after validation is wrapped in a try/catch that swallows every failure -- including email send failures -- because the response message must be byte-identical regardless of what happened, to prevent account enumeration via the forgot-password form.
- `resetPassword` (`lib/auth/password-reset.ts`) validates the token (found, unused, unexpired) and new password (8+ characters), updates the password hash, marks the token used, invalidates any sibling outstanding tokens for the same user as defense-in-depth, revokes every active `Session` for the user (mirrors `clearSession()`'s revocation in `lib/auth/session.ts` -- a reset should force re-authentication everywhere), and writes a `PASSWORD_RESET_COMPLETED` audit log.
- Added `app/reset-password/[token]/page.tsx` (looks up the token via `lookupPasswordResetToken`, shows an "invalid or expired" state or the reset form), `ResetPasswordForm.tsx`, and `actions.ts` (`resetPasswordAction`, bound to the token like `acceptInvitationAction`). Unlike invitation acceptance, a successful reset does **not** auto-create a session -- it redirects to `/login?reset=success`, requiring the user to prove the new password immediately via a normal login. `app/login/page.tsx` now reads that query param to show a one-line success confirmation.
- Added `tests/password-reset.test.ts` (23 tests: token issuance for a real/active user, silent no-op for an unknown or `DISABLED` email, email-case normalization, invalidation of earlier outstanding tokens on a new request, audit-log skip when the user has no active membership, token lookup valid/unknown/expired/used states, and `resetPassword` success plus every rejection path -- short password, unknown/expired/already-used token, inactive user, sibling-token invalidation). Renamed `tests/invitation-tokens.test.ts` to `tests/security-tokens.test.ts` to match the utility's new generic scope.
- Verified end-to-end against a live dev server and Postgres (Playwright): forgot-password shows an identical message for a real vs. nonexistent email, the reset link updates the password and revokes all sessions, the new password works for a fresh login, and both an unknown token and a replayed (already-used) token are rejected by the `/reset-password/[token]` page.
- Backfilled `docs/security/AUTHENTICATION_AND_AUTHORIZATION.md`, `docs/architecture/ARCHITECTURE_OVERVIEW.md`, `docs/architecture/API_DESIGN.md`, `docs/features/USER_FLOWS.md`, `docs/financial/AUDIT_LOG_POLICY.md`, `docs/planning/DEVELOPMENT_PLAN.md`, and `docs/planning/FEATURE_ROADMAP.md`, which still described both invitations (Phase 9A) and password reset as unimplemented/deferred/placeholder.
- Implemented Phase 10A: replaced the `RECONCILIATION_SUMMARY` report's match-count table with a true financial reconciliation report, entirely by composing existing services rather than duplicating their arithmetic. `lib/reports/generation.ts`'s `buildFinancialReconciliationReportTable` calls `calculateReconciliationTieOut` (`lib/reconciliation/tie-out-summary.ts`) and `evaluateApprovalReadiness` (`lib/reconciliation/approval-validation.ts`) directly, per reconciliation run overlapping the report period, for every figure those services already compute (bank/ledger totals, matched/unmatched/exception amounts, variance, outstanding counts, approval readiness). `ReportGenerationDatabase` was widened to compose `ApprovalValidationDatabase` (itself composing `TieOutSummaryDatabase`) instead of declaring its own narrower, overlapping shape.
- The new report adds three things neither existing service computes: total deposits/withdrawals (a `transaction.aggregate` on `creditAmount`/`debitAmount`, since tie-out only sums the net `amount`), an adjustment count (`transactionAdjustment.count`, scoped the same way as the tie-out's bank/ledger legs), and an "Approval Summary" string assembled from the run's status plus (for `APPROVED`/`REOPENED` runs) the `RECONCILIATION_RUN_APPROVED` audit log's `approvalReason`/`overrodeOutstandingItems` metadata already written by `approveReconciliationRun` (`lib/reconciliation/run-lifecycle.ts`) -- no new approval-reason storage was added. "Opening balance" is always reported as `0.00`: this system has no absolute running-balance field anywhere in the schema, so "closing balance" is the period's net movement, which is exactly the tie-out's bank/ledger transaction total (since `amount = creditAmount - debitAmount` for every transaction, per `lib/imports/csv-core.ts`/`lib/transactions/adjustment.ts`) -- reported, not recomputed.
- Each report row now covers: organization, bank account, reporting period, run status, prepared-by/approved-by (`User.fullName`, resolved from `ReconciliationRun.createdBy`/`approvedBy`), preparation/approval dates, a financial section (opening balance, total deposits, total withdrawals, bank/ledger closing balance, variance, matched amount, unmatched bank/ledger amount, exception amount), and an operational section (outstanding exceptions, outstanding transactions, adjustment count, approval summary). CSV export (`lib/reports/csv-writer.ts`) and the report viewer (`app/dashboard/reports/page.tsx`) required no changes beyond a wider table and an updated type label ("Financial reconciliation report"), since both already render an arbitrary `{headers, rows}` table.
- Did not implement PDF/XLSX export: the existing architecture (`lib/reports/csv-writer.ts` serializing a generic `ReportTable` to CSV text, written via plain `fs.writeFile`) has no PDF/XLSX rendering capability to naturally extend, and none was added.
- Rewrote `tests/reports-generation-core.test.ts`'s `RECONCILIATION_SUMMARY` coverage (8 tests: period scoping, empty-period short-circuit that skips all lookups, full single-run row assembly, missing bank-account/user lookups falling back to the raw ID, all four approval-summary variants, and multi-run independence) against a combined mock database that reuses the same where-clause-inspection pattern as `tests/reconciliation-tie-out-summary.test.ts` and `tests/reconciliation-approval-validation.test.ts`. `EXCEPTION_LIST`/`UNMATCHED_TRANSACTIONS`/validation tests are unchanged.

All arithmetic uses `Prisma.Decimal` end-to-end (no floating-point), and bank/ledger/unmatched/exception amounts are scoped to the run's period via `transactionDate`, mirroring how `lib/reports/generation.ts` scopes unmatched and exception transactions to a report period. ## 2026-07-09

### Added

- Implemented the reconciliation run lifecycle workflow: draft/in_progress/reopened runs can be submitted for review, and ready-for-review runs can be approved (`lib/reconciliation/run-lifecycle.ts`).
- Added server-side permission enforcement for reconciliation actions: `reconciliation.run` for manual matching and submission for review, `reconciliation.approve` for run approval.
- Added audit log events for reconciliation run submission (`RECONCILIATION_RUN_SUBMITTED`) and approval (`RECONCILIATION_RUN_APPROVED`).
- Hardened reconciliation concurrency: atomic transaction-status claims prevent duplicate confirmed matches, and compare-and-swap (CAS) run status transitions keep manual matching, submission, and approval consistent under concurrent requests.
- Implemented reconciliation match correction (`correctManualMatch`): a confirmed match can be corrected by replacing one side (bank or ledger transaction) with a required reason; the original match becomes `removed` and a new `confirmed` match is created linked via `correctedFromMatchId`. Audit-logged as `RECONCILIATION_MATCH_CORRECTED`.
- Implemented reconciliation run reopening (`reopenReconciliationRun`): an `approved` run can be reopened by a user with `reconciliation.approve` and a required reason, recording `reopenedBy`/`reopenedAt` while preserving prior approval history (`approvedBy`/`approvedAt`/`completedAt` are not cleared). Audit-logged as `RECONCILIATION_RUN_REOPENED`.
- Completed Phase 6D concurrency hardening: `removeManualMatch` and `correctManualMatch` previously re-checked the parent run's lock status with a plain read (`findUnique`) before mutating match state, leaving a window where a concurrent `submitReconciliationRunForReview` or `approveReconciliationRun` could transition the run to `ready_for_review`/`approved` after the check passed but before the match write committed. Both functions now share an `assertRunEditable` helper that performs a status-preserving CAS `updateMany` on the run row immediately before the mutation, so the check and the mutation serialize atomically against concurrent lifecycle transitions. Added regression tests covering both functions failing with `CONFLICT` when a concurrent submission wins the race (`tests/reconciliation-manual-match.test.ts`, `tests/reconciliation-match-correction.test.ts`).
